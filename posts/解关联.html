关联子查询

《Unnesting Arbitrary Queries》 这篇文章主要是HyPer他们针对任意的关联查询场景，提出了一种通用解关联的办法。

通用解关联方式算法更为复杂，改写完的潜在计划执行成本会比简单的按照特定模式下的解关联代价要更高。所以解关联的思路分成两步，1. 针对常见的Pattern 直接按照简单解关联转化 2. 解决不了的，尝试更通用的推导方式。

背景

SQL-99标准允许在查询的几乎所有位置使用关联子查询。从用户的角度来看，关联子查询可以极大地简化复杂 SQL 的书写。然而，与外部查询相关的嵌套查询往往会导致 dependent joins ，并采用nestloop执行，从而导致性能不佳，详细说明见上一篇文章。目前大多数系统采用启发式方法去做解关联，虽然这些方法均可以显著地加快查询处理速度，但通常仅限于某些类型的查询。

论文对上面改写成普通Join的情况做了个总结。上述我们说过的简单解关联技术依赖于一个特殊条件，**子查询中用到的属性值（比如场景1中的e2.id）可以用来替换外层查询确定的自由变量(s.id)**。但在这里，情况并非如此，例如，s.year 不能被替换。没有精确理解这句话含义，直接把原文附上了。

To the best of our knowledge, no existing system can unnest such a query. And indeed, unnesting this query is hard: Standard unnesting techniques rely upon the fact that at- tributes available within the query can be used to substitute the free variables determined by the outer query. This is not the case here, s.year for example cannot be substituted.

​ 针对上面这两种场景，特别是第二种这种包含复杂的条件和子查询结构时，直接等价替换成另一种SQL会比较难以证明其逻辑正确性，比如直接替换成非等值Join在算AGG是否能直接等价，需要花一些时间去做论证和例举了。所以个人认为通用的方法和关系代数推导在这儿就显得尤为重要了。

​ 所以其意思是，第一步先用去 T1 重过的数据集合 和 T2进行关联计算，第二步是用集合T1再和关联查询的结果做一次普通Join，需要保证集合T1的结果”正常”。

![image-20240927215048287](/Users/yifei/Library/Application Support/typora-user-images/image-20240927215048287.png)

按照上面这个变化，场景1中对应的计划图变化如下，等价规则让我们将最佳成绩的计算限制在每个学生身上，而不是对每一对 (student, exam) 进行重复计算。因此，dependent joins 仅在学生的ID投影上执行。

从某种程度上说，这构成了从外部（左）连接参数向内部（右）参数的侧面信息传递，从而进一步去重且降低执行开销。因此，重要的是要按照真正消除重复的语义来实现Projection，而不是按照SQL中保留重复的语义来实现。

这里最最重要的事情就是将一个普通的dependent joins改造成一个普通Join和一个去重后的multi-set的dependent joins, 这非常有利于在后续的等价变化能够进一步做下推和去除关联查询。

第二步. 依赖Join下推

目标： 依赖连接下推的最终目标是达到这样一个状态：右侧不再依赖于左侧，即

![image-20240927233519856](/Users/yifei/Library/Application Support/typora-user-images/image-20240927233519856.png)

  • 下推过选择 (σ)：直接先算里面的JOIN，如下。
    $$
    D1 ⋈ σp(T2) = σp(D1 ⋈T2 ) // ⋈指的仍然是 dependent-join
    $$

  • 下推过连接（⋈）：如果 T2 本身包含连接，需要根据Join的类型（Inner、left等）和具体条件来决定如何下推。

    • InnerJoin 的变化规则如下。

      • 如果依赖连接产生的值仅在T1或者T2的一侧，那么可以将这个依赖连接推到对应的那一侧。这意味着，如果只有一侧的查询逻辑依赖于子查询的结果，那么就只在那一侧进行连接操作。
      • 如果依赖查询的两侧都需要，那么需要在两边都复制这个依赖连接，然后再进行一次普通Join。请注意，这种推下规则过于悲观，如果我们将依赖连接推到两边，我们必须增加连接谓词，以便两边都匹配D值。但是请注意，相对于原始表达式，T1和T2都至少被执行了|D|次。

      ![image-20240929154950122](/Users/yifei/Library/Application Support/typora-user-images/image-20240929154950122.png)

    • OuterJoin 和 AntiJoin

      • 这两个主要区别在于LEFT和RIGHT的时候，如果是Null 侧的话就不能分开，结果会错误，这里需要小心。

      ![image-20240929160035028](/Users/yifei/Library/Application Support/typora-user-images/image-20240929160035028.png)

  • 投影推下 (π)：如果 T2*上有投影，可以推到依赖连接的两边，计算之后把投影的值加上就可以。

  • 聚合推下 (γ)：如果 T2 上有聚合，需要确保依赖连接产生的属性在进行分组操作时得以保留。

    聚合函数不变，GrouBy需要加上T2依赖的列属性值。改造完需要考虑nullable属性等是否能够满足，论文中没有说明，会影响正确性。

​ 通过上述这些转换，每个依赖连接要么在某个点通过替换被消除，要么最终在基础关系之前（or ends up in front of a base relation ），在这种情况下它可以被转化为非依赖连接。因此，依赖连接可以从任何查询中消除。

P.S 没有完全理解这个 in front of a base relation 的具体含义, 在基础的表上的 depenant joins 转化成一个普通Join。

这种方法的一个潜在风险是 D 可能会非常大，因为它是嵌套子查询的所有变量绑定的集合。但幸运的是，现实情况并非如此。注意，如果原始的嵌套连接是T1 ![image-20240929170242498](/Users/yifei/Library/Application Support/typora-user-images/image-20240929170242498.png) T2，那么 |D| ≤ |T1|。因此，如果在去相关子查询之后，最顶层的连接是一个将T1存储在哈希表中的哈希连接，那么计算D的内存消耗最多是原来的两倍，这就是最坏的情况。如果我们知道T1中的值是无重复的，例如因为它们包含一个主键，我们甚至可以避免实现D，而改为读取连接哈希表，从而消除其他解关联带来的开销。在正面，我们将一个O(n^2)的操作转化为了一个（理想情况下的）O(n)操作，这非常值。

P.S. 这里的潜在问题是D能够被多次复用，如果不能的话，仍然会带来比较大的消耗。

更多优化

simple unnesting时，它完全消除了查询中的任何关联查询，得到的逻辑计划看起来就好像查询是在没有关联子查询一样。然而，general unnesting的情况必须添加计算域D的投影，以及与D的连接，这会导致一些额外的成本。当然，计算D通常仍然比计算嵌套子查询更可取，因为D的计算和与D的连接只造成一次性成本，而嵌套评估则会导致二次方的运行时间。但尽管如此，完全消除D还是很诱人的，有时也是可能的，如前面示例查询Q1中所见（参见图8）。

  • 替代 D

一般来说，如果我们可以用子树中已经存在的值来替代D，我们就可以消除D。这通常在等值连接的情况下是常见的，例如查询包含了表达式D ![image-20240929170242498](/Users/yifei/Library/Application Support/typora-user-images/image-20240929170242498.png) D.a=R.b R，我们可以通过检查R.b的值来了解D.a的可能值，这些值可以到达原始的依赖连接。当然D可能包含在R中不存在的值，但这些值永远不会找到join-pair ，因此也永远不会到达原始的依赖连接，因此我们可以忽略它们。

为了决定替代，我们首先必须分析查询树，以找到由连接和过滤条件所诱导的等价类。例如，过滤条件σa=b意味着a和b处于同一个等价类中。我们知道在最终结果中a和b具有相同的值，因此我们可以提前用b替代a，计算这些等价类是相对直接的。

一个潜在的问题来源可能是外连接，它可能导致上述例子中的a和b不相等，但由于最顶层对D的连接是已知的不需要NULL值的时候仍然可以桌替换 。

在确定了等价类C之后，我们可以根据以下所示决定可能的替代方案：
$$
D⋈T⊆χ A(D):B(T)
if ∃ B⊆A(T):A(D)≡ CB
$$

因此，我们不是与D进行连接，而是可以扩展T（使用map运算符），并通过使用等价属性来计算D中隐含的属性值。请注意，这只是因为D是一个集合才成立的。注意，替代可能会增加中间结果的大小，两种表述之间的关系不是等于（=）而是包含于（⊆）。这种基数增加是由于失去了与D连接的（潜在的）剪枝能力。我们不是用T中的每个元组来评估形成的相关子树，这些元组将在D中找到一个连接伙伴，而是用T中的每个元组来评估。那些没有连接伙伴的元组（只会）在树的更高层次上被消除，也就是在执行原始的依赖连接时，但中间结果可能会更大。

因此，只有当与D的连接是不选择性的时，替代才有利。在查询Q1中就是这种情况，因此使用替代是一个好的选择，但一般来说，查询优化必须比较两种选择的代价并选择更优的一个。对于选择性连接，最好是保留D，从而尽可能早地消除元组。
