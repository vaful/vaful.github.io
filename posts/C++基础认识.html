从一个JAVA开发者转 C++开发，经常听起来会比较难，所以畏惧。本文是我从一个JAVA开发者的视角学习 <<C++ High Performance>> 这本书之后的应对C++的基础开发，针对和Java 一些不同点，做了一个总结。个人感觉有个重点是C++程序员需要时刻关注类的生命周期，内存分配和回收这部分相关内容。

  1. Value Vs Reference

    C++ 支持值语value semantics and reference semantics。值语义使我们可以通过值传递对象，而不仅仅是传递对象的引用。在 C++ 中，值语义是默认的，这意味着当你传递一个类或结构体的实例时，它的行为与传递一个 int、float 或其他基本类型是一样的。

    要使用引用语义，我们需要显式地使用引用或指针。C++ 的类型系统允许我们明确地声明对象的所有权(Ownership) 。

  2. 对象所有权

    除了非常罕见的情况外，C++ 程序员应该将内存管理交给容器和智能指针，而不是依赖手动内存管理。

    明确地说，通过使用 std::shared_ptr 来管理每个对象，几乎可以在 C++ 中模拟 Java 的垃圾回收模型。需要注意的是，垃圾回收语言和 std::shared_ptr 使用的分配跟踪算法并不相同。std::shared_ptr 是基于引用计数算法的智能指针，该算法在对象存在循环依赖时会导致内存泄漏。而垃圾回收语言则有更复杂的方法来处理和释放循环依赖的对象。

    然而，与其依赖垃圾收集器，强制严格的所有权管理能够巧妙地避免由于默认共享对象而导致的细微错误，这在 Java 中是常见的情况。

    如果程序员在 C++ 中尽量减少共享所有权，生成的代码将更易于使用且更难被滥用，因为这种严格的所有权模型能够强制类的用户按照预期方式使用该类。

  3. C++中的确定性析构

    在C++中，对象的析构是确定性的。这意味着我们（可以）确切地知道对象何时被销毁。这与像Java这样的垃圾回收语言不同，在这些语言中，垃圾回收器决定何时对无引用的对象进行最终处理。

    在C++中，我们可以可靠地逆转对象生命周期内所完成的操作。乍一看，这似乎只是一个小细节。但实际上，它对我们如何提供异常安全保障和处理资源（如内存、文件句柄、互斥锁等等）有着深远的影响。

    确定性析构也是使C++具有可预测性的一项重要特性。这在程序员中备受推崇，并且是性能关键型应用的一个要求。

  4. C++的缺点

    在比较C++与其他编程语言时，如前所述，C++有更多的概念需要学习，因此正确使用C++并充分发挥其潜力较为困难。然而，如果程序员能够掌握C++，那么较高的复杂性会转化为优势，代码库将变得更健壮，性能也会更好。

    尽管如此，C++还是存在一些无法避免的缺点。其中最严重的缺点包括长时间的编译时间和复杂的库导入机制。在C++20之前，C++一直依赖于一个过时的导入系统，导入的头文件只是简单地粘贴到包含它们的文件中。C++20引入的模块系统（Modules）将解决一些基于头文件包含机制的问题，同时也会对大型项目的编译时间产生积极影响。

    另一个明显的缺点是C++提供的库较少。虽然其他语言通常自带各种应用所需的大部分库，例如图形、用户界面、网络、多线程、资源管理等，但C++提供的库基本上仅限于最基本的算法、多线程处理和从C++17开始的文件系统处理。对于其他功能，程序员不得不依赖外部库。

      • auto &&

        • Forwarding Reference：在函数模板中，T&& 和 auto&& 是转发引用，能够绑定左值和右值，并保持原始值类别。这对于具有传递和完美转发的泛型函数非常有用。

        作者建议对于基本类型（如 int、float 等）和较小的非基本类型（如 std::pair 和 std::complex），使用 const auto。对于可能复制代价较高的大型类型，使用 const auto&。

        auto& 和 auto 则应仅在需要可变引用或显式复制行为时使用；这样可以向阅读代码的人传达这些变量的重要性，因为它们要么复制一个对象，要么修改一个引用的对象。

        最后，auto&& 仅用于转发代码。

    2. Move 语义（the rule of five and rule of zero）

      ![image-20240705154730049](/Users/yifei/Library/Application Support/typora-user-images/image-20240705154730049.png)

      分配和拷贝是缓慢的过程，而且在许多情况下，源对象已经不再需要。通过使用Move语义，编译器可以检测到类似这些旧对象不再与变量绑定的情况，并执行移动操作来取代拷贝操作。

      ![image-20240705154939706](/Users/yifei/Library/Application Support/typora-user-images/image-20240705154939706.png)

      要完全理解移动语义，我们需要回到 C++ 中类和资源获取的基础概念之一。

      在 C++ 中的一个基本概念是，类应该完全管理其资源。这意味着，当一个类被复制、移动、赋值复制、赋值移动或销毁时，类应该确保其资源被相应地处理。实现这五个函数的必要性通常被称为“五法则”（The Rule of Five）。

      在这个例子中，类 Buffer 中处理的资源是在构造函数中分配的一块内存。内存可能是类处理的最常见的资源，但资源可以有很多种，比如互斥锁（mutex）、显卡上的纹理句柄、线程句柄等等。

      五法则（The Rule of Five）中提到的五个函数包括：复制构造函数、复制赋值运算符、移动构造函数、移动赋值运算符和析构函数。接下来，我们将首先介绍复制构造函数、复制赋值运算符和析构函数，这些函数都需要参与资源管理。

      虽然正确实现上面这三个函数（复制构造函数、复制赋值运算符和析构函数）已经足以让一个类管理其内部资源，但仍然有两个问题：

      1. 无法复制的资源：在 Buffer 类的例子中，我们的资源可以被复制，但是存在其他类型的资源是无法复制的。例如，一个类内部包含的资源可能是 std::thread、网络连接或其他不能复制的资源。在这些情况下，我们无法在对象之间传递资源。
      2. 不必要的复制：如果我们从一个函数返回 Buffer 类的实例，整个数组需要被复制。虽然编译器在某些情况下会优化掉这次复制操作。

      这些问题的解决方案是使用移动语义。除了复制构造函数和复制赋值运算符之外，我们可以为类添加移动构造函数和移动赋值运算符。但是他们不接受 const 引用（const Buffer&）作为参数，而是接受 Buffer&& 类型的右值引用对象。

      正如你所看到的，这些函数不会抛出任何异常，因此可以标记为 noexcept。这是因为，与复制构造函数和复制赋值运算符不同，它们不分配内存或执行可能抛出异常的操作。

      • **noexcept **：通过标记 Move构造函数和Move赋值运算符为 noexcept，告诉编译器这些操作不会抛出任何异常。这能让编译器在进行某些优化时更有信心，比如在某些标准容器操作中更倾向于移动而非复制。
      • 与复制操作的区别：复制构造函数和复制赋值运算符必须分配新内存，这可能会抛出 std::bad_alloc 异常，因此通常无法标记为 noexcept。

      使用移动语义和 noexcept 能确保你的类在管理资源时既高效又安全。这是现代 C++ 编程中至关重要的最佳实践之一。

      • Named variables and rvalues

        编译器在何时允许移动对象而不是进行拷贝呢？简短的回答是：当对象可以被归类为右值（rvalue）时，编译器会移动该对象。

        右值这个术语可能听起来有些复杂，但本质上，它只是一个没有与命名变量绑定的对象：

        1. 临时对象（Temporary Objects）：这些对象在表达式求值后立即销毁。例如，函数返回的临时对象或表达式结果。
        2. 字面值常量（Literal Constants）：直接出现在代码中的常量，如 5、3.14、字符串字面值等。这些常量没有明确的命名变量绑定它们。
        3. 使用 std::move 函数：通过显式调用 std::move，可以将一个左值强制转换为右值引用，这表明我们不再需要该对象，编译器可以安全地移动它。

        在 C++ 中，移动语义依赖于右值引用（rvalue references）来实现对象的高效转移而不进行拷贝。因为右值引用通常是在编译器确定对象不再被需要时才进行的，并且这种对象不应再被使用。

      • Rule Of Zero

        在实践中，你很少需要自己编写复制/移动构造函数、复制/移动赋值运算符和析构函数。将类设计成不需要显式编写这些特殊成员函数（或者仅使用默认声明）通常称为零法则（The Rule of Zero）。这意味着，如果应用程序代码库中的一个类需要显式编写这些函数，那么这部分代码可能更适合放在库代码中。

        当使用默认生成的移动赋值运算符时，有一个常见的陷阱：类中混合了基础类型和更高级的复合类型。与复合类型不同，基础类型（如 int、float 和 bool）在被移动时只是简单地被复制，因为它们不管理任何资源。这可能会导致部分资源没有正确移动，从而引发问题。
